不想写报告了，遂来写日寄，结果日寄刚写到一半（上面引用那里），冰岩的集群爆了，于是赶紧过去修集群

2c2g 的最小的机器卡住了，导致整个集群 api 不可用，都 504 了。简单重启一下机器其实就能解决

但是这个问题已经出现了若干次，之前爆炸了之后给这个结点加上了污点。这次为了更加一劳永逸的解决时不时就要重启机器的问题，我打算把这个小机器踢出集群得了

ssh 上了这个机器，很快啊，一下就把 k3s 给卸了

然后整个集群再次爆炸

我原本计划的是先给他卸了，然后重装 k3s 变成一个 worker。但是在我卸完 k3s，集群 api 不可用之后，另一个人去剩下的一个 master 上重启了一下 k3s 服务，这下完蛋了

集群的四个机器里一共有两个真 master，现在一台让我卸了 k3s，一台重启了服务。重启服务的一台服务永远起不来，因为连不上另一个 master。被我卸了的这台机器也无法重装 k3s，因为另一台 master 的服务没起来。这下彻底变成死锁了

集群里的另外两个 k3s-agent 现在也变成了不能动状态。一旦他们重启服务就再起不能了，于是集群彻底爆炸

用尽搜索大法，又和 gpt 高强度交互到半夜，依旧回天乏术。现在所有 master 的 etcd 全都是 dead 状态

折腾到了**凌晨四点半**，实在是撑不住了，明天还要考六级

集群爆炸的问题等明天有 k3s 大师再来修吧，我怕我再整出更大的爆炸，数据丢失了就真完蛋了
